# PRD — Claim Assistant Prototype

> **Что это:** упрощённый PRD для тестового прототипа, который можно запустить за выходные, открыть с телефона, дать ссылку 5–10 знакомым и собрать обратную связь по UX и качеству AI-оценки.

| | |
|---|---|
| **Версия** | 0.1 (Prototype) |
| **Базируется на** | PRD v1.0 (full) — `PRD_Claim_Assistant.md` |
| **Дата** | 22 апреля 2026 |
| **Цель** | Прогнать сценарий «Залив квартиры» на 5–10 реальных кейсах знакомых |
| **Срок сборки** | 2–3 дня (вайбкодинг с Claude Code или Cursor) |
| **Стек** | Next.js 15 + Tailwind + Anthropic API → деплой на Vercel |

## 1. Сильное упрощение по сравнению с full PRD

### Что **остаётся** в прототипе

| Функция | Зачем нужна для теста |
|---|---|
| Дерево вопросов (`question_tree.json`) | Проверить, доходит ли клиент до конца, не теряется ли |
| Камера смартфона + overlay-подсказки | Главный UX-эксперимент: помогают ли подсказки снимать лучше |
| On-device проверка резкости (Laplacian) | Проверить, что отсев плохих фото действительно работает |
| EXIF-метаданные (дата, GPS, модель) | Понять, насколько часто iOS вырезает GPS |
| Vision-классификация через Claude API | Главный AI-эксперимент: насколько точна оценка повреждения |
| Расчёт сметы по справочникам | Проверить, что цифры адекватны (±30%) и не пугают клиента |
| PDF-отчёт со сметой | Артефакт для шеринга с экспертом-знакомым на ревью |
| Скачивание JSON-пакета | Для разбора кейсов задним числом |
| Адаптив под телефон + ноутбук | Базовая юзабельность |
| **Админ-страница `/admin`** | Калибровка весов + просмотр истории + редактирование справочников |

### Что **отрезано** на этапе прототипа

| Что отрезаем | Почему | Чем компенсируем в демо |
|---|---|---|
| СберID, СБС, DaData | Это интеграции, дни работы | Простая форма «введите ФИО, телефон» |
| Vector DB с похожими кейсами | Нет архива | Захардкодим 3 примера в системный промпт LLM |
| Fine-tuned Vision-модель | Нужен датасет и обучение | Используем Claude Sonnet 4.5 vision напрямую |
| AR-рулетка (WebXR) | Не работает на iOS Safari | Просим класть монету 10₽ или карту в кадр для масштаба |
| Temporal/Kafka, микросервисы | Прототип — один Next.js | Все шаги — в одной serverless-функции |
| Multi-region кэш, K8s, observability | Не нужно для 50 кейсов | Vercel + браузер пользователя |
| Полный RBAC в админке | Слишком долго | Один пароль на `/admin` через переменную окружения |
| WCAG 2.1 AA, локализация | Не цель прототипа | Базовые ARIA-атрибуты от shadcn/ui |
| Persistence: PostgreSQL, S3, аудит-лог | Прототипу хватит браузера | localStorage клиента + Vercel KV для админ-истории |

### Сценарии: только залив квартиры (полный AI-флоу)

Для остальных типов событий (пожар/взрыв/стихийное/кража) — дерево вопросов работает, фото загружаются, **но AI-расчёт не применяется**: показывается экран «Спасибо, ваш кейс передан эксперту» и формируется JSON-пакет. Это позволит на этапе теста собрать структуру вопросов и убедиться, что флоу корректен, без необходимости размечать справочники для всех 5 типов.

| Шаг | Экран | Что в нём ключевого |

| 1 | `/` Приветствие | Кнопка «Зафиксировать страховое событие» + дисклеймер «Это прототип, данные не передаются в страховую» |
| 2 | `/flow/intro` | 7 общих вопросов из дерева, prefill по дефолту, прогресс-бар «Шаг 4 из 12» |
| 3 | Выбор типа события | Если **залив** → ветка с расчётом; если другое — упрощённое прохождение «передадим эксперту» |
| 4 | `/flow/flood` | 8 вопросов из ветки `flood` |

### Камера: главный UX-эксперимент прототипа

```
┌────────────────────────────────┐
│  ┌──────────────────────────┐  │ ← счётчик «3/10»
│  │                          │  │
│  │      ┌───────┐           │  │ ← рамка цели
│  │      │       │           │  │
│  │      │       │           │  │
│  │      └───────┘           │  │
│  │                          │  │
│  │   📐 Уровень ровный      │  │ ← гироскоп
│  │   ☀️ Освещение хорошее   │  │ ← анализ яркости
│  │   📏 Положите монету     │  │ ← подсказка масштаба
│  └──────────────────────────┘  │
│        ⭕ снять фото            │
└────────────────────────────────┘
```

**Реализация (важные технические нюансы для прототипа):**

- `<video autoplay playsinline muted>` — без `playsinline` на iOS откроется во весь экран и эксперимент сломается
- `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' } })` — задняя камера
- Снимок: `canvas.drawImage(video) → canvas.toBlob()`
- Резкость: считаем variance of Laplacian вручную по `getImageData()` (без OpenCV.js — ~80 строк JS, экономит 8MB загрузки)
- Яркость: средняя по Y-каналу того же canvas, обновляется 2 раза в секунду
- Уровень: `DeviceOrientationEvent` (на iOS требует разрешения)
- EXIF: `exifr` (50KB)

### Известные ограничения прототипа

- **iOS Safari вырезает GPS** из фото, снятых через `getUserMedia` — это политика приватности Apple, обойти нельзя. Координаты будут только если файл загружают из галереи.
- **AR-рулетка не работает** на iOS Safari (WebXR не поддерживается). Используем эталон масштаба (монета).
- **Размер фото с iPhone до 5 МБ** — ужимаем до 1600px по длинной стороне перед отправкой в Claude (экономия токенов).

## 4. AI-pipeline в прототипе: один вызов, никакой оркестрации

В отличие от full PRD (где 8 шагов pipeline через Temporal), здесь — один вызов serverless-функции, который делает всё за 8–15 секунд.

```
POST /api/analyze
{
  "context": { ...IncidentContext... },
  "photos": [
    { "base64": "...", "exif": { ... } },
    ...
  ]
}
```

**Внутри функции:**

1. Сжать фото до 1600px (на стороне сервера, через `sharp`)
2. Один вызов Anthropic API с системным промптом + контекстом + всеми фото в `image` блоках
3. Парсинг JSON-ответа от Claude
4. Применение справочников и калибровочных весов (в `lib/calculator.ts`)
5. Возврат структурированного отчёта

**Системный промпт (упрощённая версия раздела 8.4 full PRD):**

```
Ты эксперт-оценщик СберСтрахования. Тебе показывают фотографии повреждений
от страхового события «залив квартиры».

Твоя задача — для каждой фотографии определить:
- damage_class: жёлтое_пятно | отслоение_обоев | вспучивание_ламината |
                разрушение_штукатурки | плесень | без_повреждений
- surface: потолок | стена | пол | дверной_проём | оконный_блок
- material_predicted: штукатурка | гипсокартон | обои | краска | ламинат | плитка
- area_estimate_m2: оценка площади (используй reference: монета 10₽=22мм,
                    кредитная карта=85.6мм, если есть в кадре)
- severity: low | medium | high | critical
- confidence: 0..1
- sub_findings: массив строк с дополнительными наблюдениями

Затем сформируй общее заключение и список работ из справочника
(коды работ, например DEM-007, PREP-001, LEVEL-004 — переданы в контексте).

Цены НЕ выдумывай. Возвращай только коды работ и материалов и
вычисленные объёмы. Расчёт стоимости система сделает сама.

Контекст помещения: { площадь S, высота h, уровень отделки, материал стен }.

Few-shot примеры (3 закрытых кейса) приложены ниже.

Верни СТРОГО валидный JSON по схеме { ... }. Без markdown, без объяснений.
```

**Few-shot примеры** — захардкожены в `lib/prompts.ts` как массив из 3 кейсов:
1. Маленькое жёлтое пятно на потолке кухни ~0.5 м² → антигрибок + грунтовка + покраска (потолок)
2. Большое пятно с тёмным краем на стене коридора ~3 м² + следы плесени → полный цикл с штукатуркой
3. Вспучивание ламината в спальне ~6 м² → демонтаж + новый ламинат с подложкой

## 5. Расчёт сметы: код вместо облачных сервисов

Файл `lib/calculator.ts` (суть алгоритма):

```typescript
function calculate(claudeOutput, context, calibration, catalogs) {
  // 1. Параметры поверхностей по разделу 9.2 full PRD
  const S = context.affected_area_m2
  const h = calibration.default_ceiling_height_m
  const P = 2 * Math.sqrt(S * 1.2)        // приближённый периметр
  const S_walls = P * h - 2.0              // минус один проём 2м²

  // 2. Объёмы работ из claudeOutput.recommended_works
  const works = claudeOutput.recommended_works.map(code => {
    const work = catalogs.works.find(w => w.code === code)
    const volume = computeVolume(code, S, S_walls, P, claudeOutput)
    const regionCoef = catalogs.regions[context.region].works_coefficient
    const finishCoef = calibration[`finish_${context.finish_level}_factor`]
    const unitPrice = work.base_price_rub * regionCoef * finishCoef
    return { ...work, volume, unitPrice, total: volume * unitPrice }
  })

  // 3. Объёмы материалов по нормам расхода
  const materials = computeMaterials(works, claudeOutput, catalogs.materials)

  // 4. Износ по ВСН 53-86(р) — только к материалам
  if (calibration.wear_apply) {
    materials.forEach(m => {
      const age = currentYear - context.last_renovation_year
      const wearK = Math.min(1, age / m.service_life_years)
      m.total = m.total * (1 - wearK)
    })
  }

  // 5. Точечная оценка
  const base = sum(works.map(w => w.total)) + sum(materials.map(m => m.total))

  // 6. Скидка за низкий confidence
  const lowConf = claudeOutput.average_confidence < 0.6
  const finalBase = base * (lowConf ? calibration.vision_low_confidence_discount : 1)

  // 7. Диапазон
  const sigma = calibration.range_sigma
  return {
    range: {
      min: Math.round(finalBase * (1 - sigma)),
      base: Math.round(finalBase),
      max: Math.round(finalBase * (1 + sigma))
    },
    works, materials,
    routed_to_expert: finalBase > calibration.stp_threshold_rub
  }
}
```

## 6. Справочники в репозитории

Все справочники лежат в `/data/*.json` и **бандлятся в код** при сборке. Это значит:

- При первом деплое — используются как есть.
- Когда администратор меняет справочник в `/admin/catalogs` — изменения сохраняются в **Vercel KV** (key-value хранилище).
- API-функция `/api/calculate` сначала читает KV, и только если в KV пусто — берёт из бандла.
- Можно сбросить к дефолту одной кнопкой («Восстановить из репозитория»).

### Что за справочники

| Файл | Содержание |
|---|---|
| `data/question_tree.json` | Дерево вопросов (без изменений из v1.0 PRD) |
| `data/works_catalog.json` | **35 видов работ** с ценами по 3 источникам каждый, апрель 2026 |
| `data/materials_catalog.json` | **13 материалов** с расходами, ценами, сроками службы по ВСН 53-86(р) |
| `data/region_coefficients.json` | 9 регионов с коэффициентами на работы и материалы |
| `data/calibration_defaults.json` | Дефолтные веса для калибровки |

Источники, на основе которых собраны цены (все — открытые, апрель 2026):

- **Работы:** Профи Ремонтник, Домашний Мастерок, СтройРемДизайн, Remont DMB, RemontProfi
- **Материалы:** Лемана ПРО (Леруа Мерлен), Петрович, ВсеИнструменты, Сатурн, Озон

Каждая запись в каталоге содержит поле `samples` — реальные цены, по которым было посчитано среднее.

## 7. Кабинет администратора

URL: `https://your-deploy.vercel.app/admin`
Аутентификация: один пароль через переменную окружения `ADMIN_PASSWORD`.

### Страницы кабинета

#### 7.1. `/admin` Дашборд

Простая страница с метриками за всё время:
- Всего кейсов: N
- Завершённых (дошли до отчёта): N
- Передано эксперту (превысили порог STP): N
- Средняя сумма базовой оценки: ₽
- Средний `confidence` Vision: 0.XX
- Гистограмма по типам повреждений

Все цифры — из Vercel KV (история кейсов).

#### 7.2. `/admin/calibration` Калибровка

Слайдеры + числовые поля для всех 11 весов из `calibration_defaults.json`:
- σ диапазона: ползунок 0.05 → 0.35
- Множители для 4 уровней отделки
- Скидка при низком confidence
- Порог STP (₽), порог трещины (мм), порог плесени (м²)
- Чекбокс «Применять физический износ»
- Высота потолков по умолчанию

Кнопки: «Сохранить и применить», «Сбросить к дефолту», «Экспорт в JSON».

После сохранения — все новые кейсы считаются с новыми весами.

#### 7.3. `/admin/catalogs` Справочники

Две вкладки: «Работы» и «Материалы».

В каждой:
- Таблица всех записей с inline-редактированием цены
- Кнопка «Скачать XLSX» (через библиотеку `xlsx` в браузере)
- Кнопка «Загрузить XLSX» (drag&drop, парсинг, превью изменений, подтверждение)
- Кнопка «Сбросить к версии из репозитория»

#### 7.4. `/admin/history` История

Таблица всех кейсов:
- Колонки: дата, тип события, регион, оценка (₽), статус, кнопка «Открыть»
- Фильтры по типу и статусу
- Экспорт в CSV

При клике «Открыть» — карточка кейса с фотографиями (с метаданными), JSON ответов клиента, AI-заключением, расчётом сметы, ссылкой на PDF.

## 8. Деплой на Vercel: пошагово

Для запуска прототипа после того, как код готов:

1. **Получить ключ Anthropic.** На `console.anthropic.com` → Settings → API Keys → Create.
2. **Запушить код в GitHub.** Один репозиторий `claim-assistant-proto`.
3. **На Vercel** → New Project → импортировать репозиторий.
4. **Environment Variables** добавить:
   - `ANTHROPIC_API_KEY` = ваш ключ
   - `ADMIN_PASSWORD` = любая строка
5. **Storage** → Create → KV → подключить к проекту (это автоматически добавит `KV_*` переменные).
6. **Deploy** → готово, URL вида `https://claim-assistant-proto.vercel.app`.
7. **Открыть с телефона** — камера потребует разрешения, дать.

Стоимость для теста на 50 кейсов:
- Vercel: бесплатный план (Hobby) хватает с запасом
- Vercel KV: бесплатный план — 30K запросов/мес
- Anthropic API: ~$0.05–0.10 за кейс с 5–10 фото на Sonnet 4.5 → **$3–5 на 50 кейсов**

## 9. План тестирования с 5–10 знакомыми

### Подготовка

- Развернуть прототип, получить публичный URL
- Подготовить короткую инструкцию (1 страница): «У вас был залив? Помогите проверить продукт за 15 минут»
- Договориться с одним знакомым-экспертом (если есть) на ревью отчётов

### Что собирать с каждого пользователя

| Метрика | Как мерить |
|---|---|
| Дошёл ли до конца | По логам в Vercel KV |
| Сколько фото сделал | По JSON кейса |
| Сколько фото отклонено валидацией | По счётчику в локальном состоянии |
| Время прохождения | Timestamp начала и конца |
| CSAT | Спросить лично: «От 1 до 5, как было?» |
| Адекватность оценки | Сравнить с реальным счётом из его ремонта (если есть) |
| Какие вопросы непонятны | Голосовой опрос после прохождения |

### Гипотезы, которые тест должен проверить

1. **Камера-подсказки реально помогают** — % отклонённых фото у тех, кто видел подсказки, ниже, чем у контрольной группы (2-3 пользователя без подсказок, через скрытый параметр URL).
2. **Claude Vision адекватно классифицирует залив** — для 7 из 10 кейсов класс повреждения совпадёт с тем, что сказал бы эксперт.
3. **Диапазон оценки попадает в реальную смету** — для 5+ кейсов с известной финальной суммой ремонта, реальная сумма находится в диапазоне [min, max] прототипа.
4. **Клиент доходит до конца** — completion rate ≥ 60%.
5. **Эмпатичный тон не раздражает** — никто не пожалуется на «слишком мягкий» язык.

### Выход из теста

- Сводная таблица результатов в Excel
- 3-5 ключевых выводов для full PRD
- Список багов и непонятных мест в UX
- Решение: идти ли на полноценный продукт или нужны ещё итерации прототипа

## 10. Что НЕ делаем сознательно

- **Не подключаем СберID.** В прототипе — обычная форма «введите имя и телефон» (не сохраняем нигде, кроме браузера).
- **Не делаем PWA installable** в первой итерации. Открывается через ссылку в браузере — этого хватит.
- **Не пишем тестов.** Прототип одноразовый, тесты съедят 30% времени.
- **Не делаем красивый дизайн.** Берём дефолт shadcn/ui + 2 акцентных цвета (Сбер-зелёный + белый).
- **Не оптимизируем стоимость токенов.** На 50 кейсов экономия не имеет смысла.
- **Не выкатываем в продакшн.** На прототипе крупными буквами на главной: «Прототип — данные не передаются в страховую».
