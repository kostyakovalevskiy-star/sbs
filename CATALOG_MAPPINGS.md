## 6. Цепочка ценообразования

```
recommended_works (от Claude)
       │
       ▼
для каждого work_code:
   work.volume   = computeVolume(code, A, P, heightFactor)
   work.price    = base_price × works_coef[region] × finish_factor
   work.total    = volume × price
       │
       ▼
для каждого work → WORK_TO_MATERIALS[code]:
   material.volume = ceil(work.volume / consumption_m2_per_package)   // в упаковках
   material.price  = base_price × materials_coef[region]
   material.total  = volume × price
   if calibration.wear_apply:
       wearK = min(1, renovation_age / service_life_years)
       total = total × (1 − wearK)
       │
       ▼
base = Σ works.total + Σ materials.total
если average_confidence < 0.6:
    base = base × vision_low_confidence_discount
range = [base × (1−σ), base, base × (1+σ)]    // σ из калибровки, по умолчанию 0.15
routed_to_expert = base > stp_threshold_rub
```

Региональные и финиш-коэффициенты — [proto/data/region_coefficients.json](proto/data/region_coefficients.json) и [proto/data/calibration_defaults.json](proto/data/calibration_defaults.json).

## 7. Слабые места в текущем подходе

1. **Один материал на одну работу** — нет вариативности по бренду / классу. AI не может сказать «премиум-плитка для премиум-отделки».
2. **2 материала висят без работ** (`MAT-PAINT-02`, `MAT-LAM-02`) — попадают в каталог, но никогда не выбираются.
3. **`material_predicted` AI не передаётся в калькулятор** — AI определяет тип материала на фото, но цена в смете всегда базовая, не учитывает класс.
4. **Один объём `A` на все работы кейса** — если в фото и стена и потолок повреждены, общая площадь приближённо размазывается на всё. AI не разбивает работы пер-фото.
5. **Высота потолка влияет только на стены** — `heightFactor` применяется к `PREP/LEVEL/PAINT/WALL` ([calculator.ts:87-93](proto/lib/calculator.ts#L87-L93)). Это спорно для `LEVEL-003/005/007` (это потолочные работы) — сейчас они тоже умножаются на heightFactor, что некорректно.
